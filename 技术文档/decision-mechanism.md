# JSON Pro 开发决策机制（DeepSeek-V4-Pro 专用）

## 核心原则

### 1. 排查优先，数据说话
- 不猜测，不加假设
- 用日志输出确定根因
- 90% 确定性 = 基于日志证据，不是直觉

### 2. 决策流程

```
加日志 → 看输出 → 分析原因 → 决策
                ↓
        ┌───────┴───────┐
        ↓               ↓
    90% 确定        不确定
        ↓               ↓
    直接修复        输出分析
    （无需确认）    （等待进一步排查）
        ↓               ↓
    输出：问题      输出：
    + 原因          - 当前日志
    + 方案          - 可能原因列表
    + 修改          - 需要补充的日志
                    - 下一步排查方向
```

### 3. 确定性标准

| 确定性 | 标准 | 行动 |
|:---|:---|:---|
| **90%+ 确定** | 日志明确指向某一行代码，且排除其他可能性 | 直接修复，无需确认 |
| **60-90%** | 日志指向某个方向，但可能有其他原因 | 输出分析，等待确认 |
| **< 60%** | 日志不足，或多种可能性并存 | 补充日志，继续排查 |

### 4. 修复后必须输出

即使 90% 确定直接修复，也必须输出：

```
## 修复报告

### 问题
[具体描述]

### 根因
[基于日志的证据]
- 日志第 X 行显示：...
- 因此确定是 ... 问题

### 方案
[修改思路]

### 修改
[具体代码变更]
- 文件：X.vue
- 第 Y 行：从 ... 改为 ...

### 验证
[如何验证修复成功]
```

### 5. 需要确认的情况

以下情况**必须询问确认**，不能直接修复：

| 情况 | 原因 |
|:---|:---|
| **核心架构改动** | 影响 MemoryIndex、StreamingValidator、数据流 |
| **接口变更** | 改方法签名、返回值类型、调用方式 |
| **跨模块大量修改** | 涉及 >3 个文件，或 >50 行变更 |
| **性能策略改变** | 改缓存策略、渲染模式、异步方案 |
| **删除功能** | 删除已有组件、方法、配置项 |
| **引入新依赖** | 新增 npm 包、外部库 |

### 6. 不确定时的输出格式

```
## 排查分析（不确定，需进一步确认）

### 当前日志
[粘贴关键日志输出]

### 可能原因列表
1. [原因A] - 证据：... - 概率：60%
2. [原因B] - 证据：... - 概率：30%
3. [原因C] - 证据：... - 概率：10%

### 无法确定的原因
[说明为什么日志不足以判断]

### 下一步排查
1. 补充日志：在 X 位置加 ...
2. 测试场景：...
3. 预期输出：...

### 建议
等待用户补充日志后，再确定根因
```

## 应用到当前问题

### 当前任务：行号不一致 + 滚动条到底内容未到底

#### 排查步骤（带决策点）

```
步骤1-7：加日志，收集数据
    ↓
分析日志：
- 如果 spacer 高度 ≠ 233万*28px → 90% 确定是根因D
- 如果 end ≠ totalLines 且 start 计算溢出 → 90% 确定是根因E
- 如果 node.id 和行号有固定偏移 → 90% 确定是根因F
- 如果多种问题并存 → 不确定，输出分析
    ↓
决策：
- 单一根因且 90% 确定 → 直接修复，输出报告
- 多种可能 → 输出分析，等待进一步排查
```

#### 修复权限

| 修改范围 | 权限 |
|:---|:---|
| VirtualScroller.vue 内部逻辑 | 90% 确定可直接修复 |
| 行号计算方式 | 90% 确定可直接修复 |
| visibleRange 算法 | 90% 确定可直接修复 |
| MemoryIndex 查询接口 | **必须确认** |
| StreamingValidator 索引构建 | **必须确认** |
| 数据流架构 | **必须确认** |
