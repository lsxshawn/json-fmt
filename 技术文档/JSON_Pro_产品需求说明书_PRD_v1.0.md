## 1. 文档说明

### 1.1 文档目的
本文档定义 JSON Pro 产品的完整需求，包括目标用户、核心功能、商业模式及技术边界，作为设计、开发、测试及运营团队的统一基准。

### 1.2 术语表

| 术语 | 定义 |
|------|------|
| 大文件 | 指 10MB 以上的 JSON 文件，通常由 API 批量返回、日志聚合或数据迁移产生 |
| 流式解析 | 不一次性将文件加载为对象，而是逐字符扫描，边读边处理 |
| 虚拟滚动 | 只渲染可见区域 DOM 元素，通过复用和位移模拟长列表滚动 |
| 按需格式化 | 仅对当前视口内的文本进行语法高亮和缩进处理 |
| 纯本地处理 | 所有计算在浏览器完成，数据不上传至任何服务器 |
| 骨架屏 | 内容加载前显示的占位动画，降低用户感知等待时间 |
| JSON Schema | 描述 JSON 数据结构的规范，用于验证和文档生成 |
| Minimap | 代码编辑器的缩略图导航组件，用于快速定位 |

## 2. 产品概述

### 2.1 产品定位

JSON Pro 是一款面向开发者和数据工作者的 **大文件 JSON 可视化与处理工具**，核心解决以下痛点：

1. **大 JSON 文件打开卡顿/崩溃** — 现有工具在 10MB+ 文件下直接卡死
2. **数据安全隐患** — 在线 JSON 工具强制上传云端，企业敏感数据无法使用
3. **格式化阻塞 UI** — 一次性格式化大文件导致浏览器假死数秒
4. **缺乏协同能力** — 团队间 JSON 接口文档、数据格式靠微信/邮件传递，版本混乱

### 2.2 产品 Slogan

> **100MB JSON 秒开，纯本地处理，团队协作无忧。**

### 2.3 产品形态

- **Web 应用**：PWA，支持离线使用，可安装为桌面应用
- **浏览器插件**（远期）：直接在浏览器开发者工具中打开网络请求的 JSON 响应
- **桌面端**（远期）：Electron 封装，支持右键"用 JSON Pro 打开"

### 2.4 核心价值主张

| 维度 | 传统方案 | JSON Pro |
|------|---------|---------|
| 打开速度 | 3-8 秒卡死 | **< 1 秒首屏** |
| 数据安全 | 上传云端 | **纯本地，零上传** |
| 格式化 | 阻塞 UI | **按需异步，无感知** |
| 协同 | 微信传文件 | **实时协同编辑** |
| 价格 | 免费但广告/数据风险 | **免费基础 + 订阅高级** |

## 3. 目标用户

### 3.1 用户画像

#### 用户 A：后端开发工程师（主力用户）

- **年龄**: 25-35 岁
- **工作**: 设计 API 接口，调试批量数据返回
- **痛点**: 
  - 分页接口一次返回 50MB JSON，浏览器控制台直接崩溃
  - 需要快速定位某个字段在深层嵌套中的位置
  - 接口变更后，前端抱怨字段类型变了，但自己不知道哪里改了
- **付费意愿**: 高（公司报销，个人不敏感）
- **使用频率**: 每天多次

#### 用户 B：数据工程师/分析师

- **年龄**: 28-40 岁
- **工作**: ETL 流程，日志分析，数据质量检查
- **痛点**:
  - 每天处理 GB 级日志文件，需要快速查看结构
  - 数据格式不一致导致入库失败，需要 Schema 验证
  - 与业务团队确认字段含义，需要共享文档
- **付费意愿**: 中高（团队工具，走项目预算）
- **使用频率**: 每周多次

#### 用户 C：全栈开发者/独立开发者

- **年龄**: 22-30 岁
- **工作**: 前后端都写，经常对接口数据
- **痛点**:
  - 需要把后端返回的 JSON 转成 TypeScript 类型定义
  - 构造 Mock 数据测试前端边界情况
  - 个人项目预算有限，需要免费或低价工具
- **付费意愿**: 中（29元/月可接受）
- **使用频率**: 每周几次

#### 用户 D：运维/SRE 工程师

- **年龄**: 27-38 岁
- **工作**: 系统监控，故障排查，日志分析
- **痛点**:
  - 生产环境日志导出为 JSON，几十MB，需要快速检索错误
  - 数据敏感，绝对不允许上传第三方
  - 需要与开发团队共享排查结果
- **付费意愿**: 高（安全刚需）
- **使用频率**: 故障时高频使用

## 4. 使用场景

### 4.1 场景一：调试 API 批量返回（高频）

**角色**: 后端开发工程师  
**背景**: 分页查询接口，一次返回 10万条用户数据，约 80MB JSON  
**流程**:
1. 从 Postman/浏览器复制响应，保存为 `.json` 文件
2. 拖拽文件到 JSON Pro
3. 0.5 秒看到数据结构，搜索 `"total"` 确认总数
4. 展开 `users[0]` 查看字段结构
5. 复制路径 `users[0].profile.avatar` 发给前端

**痛点解决**:
- 以前：VS Code 打开 80MB JSON，内存占用 2GB，滚动卡顿
- 现在：秒开，流畅滚动，纯本地安全

### 4.2 场景二：分析生产日志（中频）

**角色**: 运维工程师  
**背景**: 线上故障，导出 ELK 日志，200MB JSON 格式  
**流程**:
1. 打开 JSON Pro，选择隐私模式（禁用缓存）
2. 加载 200MB 日志文件
3. 搜索 `"level": "ERROR"`，快速定位错误日志
4. 展开错误条目，查看 `stack_trace` 和 `context`
5. 复制错误路径和上下文，粘贴到故障工单

**痛点解决**:
- 以前：用 `jq` 命令行查询，不方便可视化；在线工具不敢用（敏感数据）
- 现在：本地可视化，搜索即时，安全无忧

### 4.3 场景三：接口文档协同（低频高价值）

**角色**: 后端团队负责人  
**背景**: 团队 10 人，前后端分离，接口频繁变更  
**流程**:
1. 在 JSON Pro 中导入当前接口返回示例
2. 一键生成 JSON Schema，定义字段类型和必填项
3. 邀请前端同事加入协同编辑
4. 前端在 Schema 上标注疑问（如 `"age"` 是否可为负）
5. 后端修改 Schema，前端实时看到变更通知
6. 导出为 Markdown 接口文档，发布到 Wiki

**痛点解决**:
- 以前：Swagger 配置繁琐，变更后不同步，靠微信群口头沟通
- 现在：可视化协同，版本可追溯，自动生成文档

### 4.4 场景四：数据迁移验证（中频）

**角色**: 数据工程师  
**背景**: 从 MySQL 导出数据到 MongoDB，需要验证 JSON 格式  
**流程**:
1. 导出数据为 500MB JSON Lines 文件
2. JSON Pro 分块加载，流式验证每行格式
3. 发现第 1523 行格式错误（缺少闭合括号）
4. 跳转到错误位置，查看上下文
5. 修复后重新验证，确认全部通过

**痛点解决**:
- 以前：用脚本验证，错误定位困难，大文件处理慢
- 现在：可视化验证，精确定位，分块处理不卡死

## 6. 核心功能详细说明

### 6.1 大文件秒开

**需求描述**: 用户打开 100MB JSON 文件，首屏时间 < 1 秒，全程不卡顿。

**用户故事**:
> 作为后端开发，当我调试接口返回的 80MB JSON 时，我希望秒开并流畅浏览，而不是等待 8 秒或浏览器崩溃。

**验收标准**:
- [ ] 100MB 文件从选择到首屏显示 < 1 秒
- [ ] 滚动帧率稳定在 55-60 fps
- [ ] 内存占用 < 200MB（100MB 文件）
- [ ] 浏览器标签页不崩溃

**技术方案**:
- 流式校验 + 索引构建（单次扫描 O(n)）
- 虚拟滚动（只渲染 30 行）
- 按需格式化（后台异步）
- 详见《核心技术方案》

### 6.2 纯本地处理

**需求描述**: 所有数据处理在浏览器完成，不上传任何服务器。

**用户故事**:
> 作为运维工程师，当我分析生产日志时，我需要确保敏感数据不会离开本地电脑。

**验收标准**:
- [ ] 打开浏览器开发者工具 Network 面板，无上传请求
- [ ] 断网状态下可正常使用
- [ ] 提供隐私模式开关
- [ ] 关闭页面后内存数据释放（可通过 Chrome Task Manager 验证）

### 6.3 按需格式化

**需求描述**: 只格式化当前可见区域，不可见区域保持原始文本，后台渐进处理。

**用户故事**:
> 作为开发者，当我滚动大 JSON 文件时，我希望看到的内容是格式化的，但不需要等待整个文件格式化完成。

**验收标准**:
- [ ] 可见区域（视口 + 缓冲区）在 100ms 内完成格式化
- [ ] 滚动时新进入视口的区域在 50ms 内格式化
- [ ] 不可见区域显示原始文本（灰色）
- [ ] 后台空闲时渐进格式化，用户无感知

### 6.4 智能搜索

**需求描述**: 支持键名即时搜索和值异步搜索，结果渐进显示。

**用户故事**:
> 作为数据分析师，当我在 50MB 日志中查找特定错误码时，我希望键名搜索立即返回，值搜索在后台进行并渐进显示。

**验收标准**:
- [ ] 键名搜索输入后 < 100ms 显示结果
- [ ] 值搜索通过 Web Worker 异步执行
- [ ] 搜索结果高亮显示，支持上下导航
- [ ] 点击搜索结果跳转到对应位置并展开父节点

### 6.5 JSON Schema 生成

**需求描述**: 从 JSON 数据一键生成 Schema，支持类型推断和约束检测。

**用户故事**:
> 作为后端开发，当我定义新接口时，我希望从示例 JSON 自动生成 Schema，用于前端验证和文档生成。

**验收标准**:
- [ ] 点击"生成 Schema"按钮，3 秒内生成
- [ ] 正确推断类型（string/number/boolean/object/array/null）
- [ ] 检测必填字段（所有键默认 required）
- [ ] 检测枚举值（如果值重复出现且种类 < 5）
- [ ] 导出为标准的 JSON Schema Draft 07 格式

### 6.6 协同编辑

**需求描述**: 多人实时编辑同一 JSON/Schema，支持光标显示和变更通知。

**用户故事**:
> 作为团队负责人，当我与前端同事确认接口字段时，我希望实时看到对方的修改和疑问，而不是通过微信截图来回沟通。

**验收标准**:
- [ ] 两人同时编辑，延迟 < 500ms
- [ ] 显示其他用户光标位置（不同颜色）
- [ ] 字段修改时，订阅者收到通知
- [ ] 支持评论批注（对特定节点添加评论）
- [ ] 离线时本地缓存，联网后自动同步

## 7. 非功能需求

### 7.1 性能需求

| 指标 | 目标值 | 测试方法 |
|------|--------|---------|
| 首屏时间 (100MB) | < 1 秒 | Chrome DevTools Performance |
| 滚动帧率 | 55-60 fps | FPS 计数器 |
| 搜索响应 (键名) | < 100 ms | console.time |
| 搜索响应 (值) | < 2 秒 (渐进) | 人工测试 |
| 节点展开 | < 50 ms | console.time |
| 内存峰值 (100MB) | < 200 MB | Chrome Task Manager |
| CPU 峰值 | < 70% (单核) | Chrome DevTools |

### 7.2 安全需求

| 需求 | 实现 |
|------|------|
| 零上传 | 无 XMLHttpRequest/Fetch 发送 JSON 数据 |
| 零持久化（默认） | 关闭标签页释放内存，不保存文件 |
| 可选本地缓存 | IndexedDB 需用户授权，可关闭 |

### 9.2 转化漏斗

```
曝光 (社区/SEO/口碑)
  ↓ 10%
访问官网
  ↓ 30%
试用免费版 (打开文件)
  ↓ 20%
```
