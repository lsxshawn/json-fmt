## 更新日期

## 一、问题背景与目标

### 1.1 原始问题
- 大文件（50MB+）解析时内存占用高达 500MB+
- 后台格式化时间过长（13-28秒）
- 浏览器容易崩溃
- 用户无法在解析过程中浏览内容

### 1.2 优化目标
- 将内存占用降低到 50MB 左右
- 首屏渲染时间降低到 <1秒
- 支持解析过程中滚动浏览
- 避免内存溢出风险

---

## 二、最终实现策略

### 2.1 核心架构：BlockCacheManager

```
┌─────────────────────────────────────────────────────────────┐
│                     完整数据 (233W行)                        │
├──────────┬──────────┬──────────┬──────────┬─────────────────┤
│  Block 0 │  Block 1 │  Block 2 │  Block 3 │     ...        │
│ (0-299)  │(300-599) │(600-899) │(900-1199 │                │
└────┬─────┴────┬─────┴────┬─────┴────┬─────┴─────────────────┘
     │          │          │          │
     └──────────┼──────────┼──────────┘
                ▼          ▼
         ┌────────────┐────────────┐
         │  活跃缓存  │   待回收   │
         │ (3个Block) │  (>1个)   │
         └────────────┴────────────┘
```

### 2.2 关键参数配置

```javascript
{
  blockSize: 300,           // 每个Block包含300个节点
  maxActiveBlocks: 3,      // 最多同时保持3个Block在内存
  recycleThreshold: 1       // 待回收队列超过1个时执行GC
}
```

### 2.3 虚拟滚动触发机制

```
滚动位置: 450行 (位于Block 1中间)
           │
           ▼
    ┌──────────────────────────────┐
    │ 检测到滚动位置变化           │
    └────────────┬─────────────────┘
                 │
           ┌─────┴─────┐
           ▼           ▼
    触发预加载       标记旧数据
    (Block 0,1,2)   (Block ≥3 或 ≤-1)
```

**触发条件：**
- 当滚动到 Block 中间位置时（超过 50%），触发加载下一个 Block
- 加载范围：当前 Block - 1 到 当前 Block + 1（共3个）

## 三、核心模块说明

### 3.1 BlockCacheManager.js

**位置：** `src/core/BlockCacheManager.js`

**主要方法：**

| 方法 | 说明 |
|------|------|
| `initialize(index, content, totalNodes)` | 初始化，解析JSON数据 |
| `loadBlocks(rangeStart, rangeEnd)` | 加载指定范围的Blocks |
| `loadBlock(blockId)` | 异步加载单个Block |
| `recycleBlocks(keepStart, keepEnd)` | 回收超出范围的Blocks |
| `getActiveNodes()` | 获取当前活跃的所有节点 |
| `clear()` | 清理所有缓存 |

**数据结构：**

```javascript
interface DataBlock {
  id: number;                    // Block编号
  startIndex: number;            // 起始节点索引
  endIndex: number;              // 结束节点索引
  nodes: Array<Node>;           // 节点数据
  status: 'loading' | 'loaded' | 'recycled' | 'error';
  lastAccessTime: number;        // 最后访问时间(用于LRU)
}
```

### 3.2 解析流程优化

**旧流程：**
```
文件读取 → 全量JSON.parse() → 构建完整树 → 渲染1000节点
           (500MB内存占用)     (耗时10+s)
```

**新流程：**
```
文件读取 → 流式校验(0内存) → 构建索引(小内存)
                                ↓
                    快速获取节点总数(用于Minimap)
                                ↓
                    初始化BlockCacheManager
                                ↓
                    加载Block 0,1,2 (约900节点)
                                ↓
                    渲染可见区域 (~21节点)
```

### 3.3 内存管理策略

**LRU回收机制：**
1. 当活跃Block数 > maxActiveBlocks (3) 时
2. 将超出范围的Block标记为待回收
3. 待回收队列超过阈值时，执行强制回收
4. 回收时清空节点数组，释放内存

**GC友好设计：**
```javascript
function forceRecycle(blockId) {
  const block = this.activeBlocks.get(blockId);
  block.nodes = null;           // 解除引用
  this.activeBlocks.delete(blockId);
  // 等待GC回收
}
```


**新增变量：**
```javascript

```

**初始化流程：**
```javascript
// 1. 流式校验和索引构建
const indexResult = streamingValidator.process(content);
memoryIndex.loadFromStreamingIndex(indexData, content);

// 2. 快速获取节点总数
const quickResult = parseAndCreateOptimized(content);
totalNodes.value = totalCount;

// 3. 初始化BlockCacheManager
blockCacheManager = new BlockCacheManager({
  blockSize: 300,
  maxActiveBlocks: 3,
  recycleThreshold: 1
});
blockCacheManager.initialize(memoryIndex, content, totalCount);

// 4. 设置回调并加载初始Blocks
blockCacheManager.onBlocksChanged = (nodes) => {
  visibleNodes.value = markRaw(nodes);
};
await blockCacheManager.loadBlocks(0, 2);
```

## 五、Minimap优化

### 5.1 基于索引的完整绘制

**问题：** 之前只绘制1000个可见节点，Minimap显示不完整

**解决方案：** Minimap现在接收 `memoryIndex` 和 `content`，基于索引绘制

**绘制函数：** `drawFromIndex()` in Minimap.vue

**原理：**
1. 利用索引中的键信息（keyCount、lineCount、structures）
2. 扫描原始内容确定每行类型（对象/数组/值）
3. 根据节点类型分配颜色
4. 使用采样算法绘制所有节点

### 5.2 Minimap显示条件修复

移除了 `parseStatus.value = ''` 的设置，确保Minimap在解析完成后仍然显示
