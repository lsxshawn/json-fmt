# JSON Pro 开发规范（Trae 引导文档）

## 文档优先级（Trae 必须遵循）

当 Trae 处理任何需求时，按以下优先级读取和遵循文档：

| 优先级 | 文档 | 路径 | 用途 |
|:---|:---|:---|:---|
| **P0（最高）** | ARCHITECTURE.md | docs/ARCHITECTURE.md | 模块划分、接口契约、禁止事项 |
| **P1** | PERFORMANCE_BUDGET.md | docs/PERFORMANCE_BUDGET.md | 内存/时间红线、优化 checklist |
| **P2** | STYLE_GUIDE.md | docs/STYLE_GUIDE.md | 色彩、字体、布局、交互 |
| **P3** | IMPLEMENTATION.md | docs/IMPLEMENTATION.md | 算法细节、伪代码、常见错误 |
| **P4** | DECISION_LOG.md | docs/DECISION_LOG.md | 历史决策、不可推翻的理由 |
| **P5（参考）** | 本文件 | docs/GUIDELINE.md | Trae 行为约束、思考流程 |

**规则**：P0 文档的约束不可违背，P1-P4 的约束在冲突时按优先级覆盖，P5 约束 Trae 自身行为。

---

## Trae 思考流程（强制）

收到需求后，Trae 必须按以下顺序思考，**不可跳过**：

```
1. 读取相关 P0-P4 文档（如有）
   → 理解模块边界和约束

2. 判断需求类型
   →  bug修复？新功能？风格调整？重构？

3. 检查影响范围
   →  哪些文件？是否跨模块？是否触及禁止事项？

4. 制定最小修改方案
   →  只改必要文件，只改必要代码

5. 输出修改计划（给用户确认）
   →  文件清单、行号范围、预期效果

6. 执行修改
   →  按确认后的计划精确执行

7. 验证约束
   →  是否违反 PERFORMANCE_BUDGET？
   →  是否违反 STYLE_GUIDE？
   →  是否违反 ARCHITECTURE 禁止事项？
```

## 需求类型处理规范

### 类型1：Bug 修复

**Trae 必须**：
1. 先定位具体文件和函数（不超过2个文件）
2. 只修复报错行，不改实现思路
3. 如果修复需要改 >3 行，先询问用户
4. 修复后检查是否引入新错误

**Trae 禁止**：
- 重构整个文件
- 改方法名/变量名
- 改函数签名
- 添加新依赖

**Prompt 模板**：
```
修复 [文件] 第 [X] 行的 [错误描述]。
只修改第 [X] 行附近，不改其他代码。
如果无法只改一行，先告诉我原因和需要的改动范围。
```

---

### 类型2：风格/视觉调整

**Trae 必须**：
1. 只改 CSS 变量和样式值，不动逻辑代码
2. 按 STYLE_GUIDE.md 的颜色映射替换
3. 输出"颜色替换清单"而不是完整 CSS

**Trae 禁止**：
- 改组件结构（添加/删除 DOM 元素）
- 改方法名/变量名
- 重构 Vue 模板
- 添加新组件

**Prompt 模板**：
```
调整 [组件] 的视觉风格。
只改 <style> 部分的颜色/间距/圆角，不改 <script>。
按以下映射替换颜色：
- 旧色A → 新色A
- 旧色B → 新色B
```

---

### 类型3：新功能实现

**Trae 必须**：
1. 先读 ARCHITECTURE.md 确认模块归属
2. 输出"实现计划"（文件清单+接口定义）给用户确认
3. 按确认后的计划分步实现
4. 每步完成后验证 PERFORMANCE_BUDGET

**Trae 禁止**：
- 一次性改 >3 个文件
- 改已有模块的接口（除非文档允许）
- 引入与架构冲突的实现方式

**Prompt 模板**：
```
实现 [功能]，参考 ARCHITECTURE.md 的 [模块] 部分。
先输出实现计划（文件清单+接口定义），我确认后再执行。
```

---

### 类型4：性能优化

**Trae 必须**：
1. 先读 PERFORMANCE_BUDGET.md 确认红线
2. 输出优化前后的预期指标对比
3. 优化后实际测量，确认达标

**Trae 禁止**：
- 以牺牲可读性为代价的极端优化
- 违反架构约束的优化（如为了快而 JSON.parse）
- 未验证就声称"优化完成"

**Prompt 模板**：
```
优化 [模块] 的 [性能指标]。
当前：[X]，目标：[Y]，参考 PERFORMANCE_BUDGET.md。
先输出优化方案，我确认后再执行。
```

## 文件修改约束

### 白名单机制

用户未明确提及的文件，**默认不可修改**。Trae 必须：

```
用户说："改按钮颜色"
Trae 想：只改 Button.vue 的 <style>
Trae 不想：改 App.vue 的布局、改 Sidebar.vue 的宽度
```

### 修改范围声明

每次修改前，Trae 必须声明：

```
本次修改涉及文件：
- src/components/Button.vue（颜色值）
- src/styles/variables.css（CSS变量）

不涉及文件：
- App.vue, Sidebar.vue, TreeView.vue 等
```

### 紧急制动

遇到以下情况，Trae 必须**停止修改并询问**：

| 情况 | Trae 行为 |
|:---|:---|
| 需要改 >3 个文件 | "需要改4个文件，是否继续？还是拆步骤？" |
| 需要改方法签名 | "需要改函数参数，可能影响调用方，确认吗？" |
| 与 ARCHITECTURE 冲突 | "当前方案与架构文档冲突，建议改为X，确认吗？" |
| 可能超性能预算 | "优化后可能超内存红线，建议改为Y，确认吗？" |
| 用户说"只改颜色"但涉及逻辑 | "颜色改动需要改状态变量，确认吗？" |

## 上下文管理规范

### 读取项目文档

Trae 在开始工作前，**应该**读取以下文件（如果存在）：

```
1. docs/ARCHITECTURE.md → 理解架构
2. docs/PERFORMANCE_BUDGET.md → 理解性能约束
3. docs/STYLE_GUIDE.md → 理解视觉规范
4. README.md → 理解项目结构
5. package.json → 理解依赖
```

### 上下文截断处理

当项目文件过多，Trae 上下文不足时：

```
Trae 必须：
1. 告诉用户"上下文不足，只能看到部分文件"
2. 询问"是否需要我重点关注某几个文件？"
3. 不要基于不完整信息猜测实现

Trae 禁止：
- 假装看过全部文件
- 基于猜测生成代码
- 不告知用户上下文截断
```

## 输出格式规范

### 修改计划格式

```
## 修改计划

### 涉及文件
- src/components/X.vue（改颜色值）
- src/styles/variables.css（新增变量）

### 不改文件
- App.vue, Sidebar.vue（与本次无关）

### 具体改动
1. X.vue 第45行：#4CAF50 → #059669
2. X.vue 第67行：padding 20px → 28px
3. variables.css 新增 --accent-soft

### 预期效果
- 字符串颜色从亮绿变柔和绿
- 层级缩进加大，呼吸感增强

### 风险
- 无（只改样式值）

请确认后执行。
```

### 执行后报告格式

```
## 执行完成

### 已修改
- src/components/X.vue（3处颜色值）
- src/styles/variables.css（新增1个变量）

### 验证
- [x] 未改方法名/变量名
- [x] 未改组件结构
- [x] 符合 STYLE_GUIDE.md 色彩系统
- [x] 未超 PERFORMANCE_BUDGET

### 截图建议
请检查 X.vue 的字符串颜色是否变为柔和绿。
```

## 常见错误预防

| 错误 | 预防方法 |
|:---|:---|
| 重构不该动的文件 | 白名单机制，只改用户提及的文件 |
| 改方法名导致引用断裂 | 禁止改方法名，除非用户明确要求 |
| CSS改动连带改逻辑 | 明确区分<style>和<script>修改范围 |
| 颜色不统一 | 强制引用 STYLE_GUIDE.md 变量，不硬编码 |
| 性能优化过头 | 优化后必须测量，对比 PERFORMANCE_BUDGET |
| 上下文不足瞎猜 | 主动告知截断，询问重点文件 |

## 用户指令优先级

当用户指令与文档冲突时：

```
用户说："不管架构文档，直接 JSON.parse"
Trae 行为：
1. 提醒用户"这与 ARCHITECTURE.md 冲突，可能导致内存爆炸"
2. 解释后果"50MB文件会从130MB涨到500MB"
3. 提供替代方案"建议用 StreamingValidator，一样快"
4. 如果用户坚持，执行但标注"违反架构约束"
```

**原则**：文档是"强烈建议"，用户是最终决策者，但 Trae 必须让用户知道风险。
