## 2026-05-09

### 决策1：不 JSON.parse 大文件

**决策**：所有 >1MB 的 JSON 文件，必须通过 StreamingValidator 流式扫描，禁止 JSON.parse

**原因**：
- JSON.parse 内存膨胀 5-6x，50MB 文件占 300MB+ 对象树
- 扁平化节点再占 200MB，总计 500MB+

**不可推翻**：
- 产品核心卖点是"大文件秒开"，JSON.parse 直接违背
- 50MB 文件 500MB 内存，无法优化到目标 130MB

---

### 决策2：浅色扁平 Web 风格

**决策**：从 IDE 深色风格改为浅色扁平 Web 风格，参考 Notion/Figma/Linear

**原因**：
- 用户需要 SaaS 产品感，不是开发工具
- IDE 风格（暗金/青绿/紧凑行高）与高级产品定位不符

**不可推翻**：
- 目标用户是后端开发者/数据工程师，他们每天面对 IDE，JSON Pro 是"工具中的工具"，需要视觉区分
- 浅色风格更符合 Web 应用心智模型

---

### 决策3：控制台改为状态光点

**决策**：删除底部日志面板，改为状态栏单个光点

**原因**：
- 用户打开 JSON 是为了看内容，不是看解析过程
- 5行步骤+耗时制造信息焦虑
- "1504ms"标黄像报警

**不可推翻**：
- 99% 用户不关心解析过程，只关心结果
- 光点颜色=状态，不需要文字翻译，符合 Apple/Linear 极简美学

---

### 决策4：Minimap 必须全局绘制

**决策**：Minimap 基于完整索引绘制全部数据，不是只渲染可见区域

**原因**：
- Minimap 是"地图"不是"望远镜"，失去全局就失去导航意义
- 性能用 Canvas 像素操作解决，不是用虚拟滚动解决

**不可推翻**：
- "只绘制可见区域的 minimap" 就是第二个主视图，完全失去产品差异化
