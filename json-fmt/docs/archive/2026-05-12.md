## Part 1: 微速滚动晃动 — 9 轮排查最终根因 + 反思

### 问题现象

鼠标滚轮微速滚动时，界面晃动不止。控制台日志确认 `startIndex` 来回震荡（56→57→56→58→57），不是正常的单向递增。

### 排查时间线（9 轮，跨两天）

| 轮次 | 日期 | 方向 | 结果 | 经验教训 |
|:--|:---|:---|:---|:---|
| 1 | 0510 | displayScrollLine 浮点 roundtrip | ⚠️ 部分有效 | `floor(localScrollTop/scaleFactor/28)` 改为 displayScrollTopRef 读 DOM scrollTop |
| 2 | 0510 | ResizeObserver 强制回流 | ⚠️ 无效 | 删除 ResizeObserver，不影响晃动 |
| 3 | 0510 | BlockCacheManager 每帧重建数组 | ⚠️ 部分有效 | `if (needsUpdate) onBlocksChanged`，减少 Vue 全量重渲染 |
| 4 | 0510 | handleNeedNodes 重复请求 | ⚠️ 无效 | `_lastReqStartBlock/_lastReqEndBlock` 守卫，晃动仍在 |
| 5 | 0510 | `.tree-nodes` GPU 加速 | ⚠️ 无效 | `will-change: transform` 不影响晃动 |
| 6 | 0512 | watcher 改为 startIndex 单值监听 | ⚠️ 无效 | 减少 onNeedNodes 调用次数但 startIndex 仍在震荡 |
| 7 | 0512 | isRangeLoaded 早退 | ⚠️ 无效 | 跳过已加载 block range 但问题在更上层 |
| 8 | 0512 | 用户反馈：startIndex 仍在 56↔57↔56 震荡 | ❌ | JS 层全部排除，问题必定在其他层 |
| **9** | **0512** | **CSS `overflow-anchor:none` + `contain:layout style`** | **✅ 根因修复** | **浏览器 Scroll Anchoring 反馈环路** |

### 最终根因：浏览器 Scroll Anchoring 反馈环路

**浏览器默认行为**：对可滚动容器启用 `overflow-anchor: auto`，当滚动锚点上方发生 DOM 变更时，浏览器自动调整 `scrollTop` 以维持视觉稳定性。

**反馈环路**：
```
scroll 事件 → localScrollTop 更新 → Vue 重渲染 offsetY/visibleItems
→ DOM 变更 → overflow-anchor:auto → 浏览器调 scrollTop
→ 新 scroll 事件 → startIndex 震荡 🔁
```

**修复**：`.tree-scroll { overflow-anchor: none }` + `.tree-content-inner { contain: layout style }`（2 行 CSS）

### 滚动残留波动修复

`offsetY` 用 `localScrollTop`（与 `startIndex` 同源），统一除以 `scaleFactor`：
- 消除 `displayScrollTopRef` 无用代码
- `skeletonTopLine` 同样统一来源

## Part 2: 虚拟展开折叠 — 从零实现

### 背景

诊断发现展开折叠三重断裂：
1. Worker `case 'toggle'` 空实现 `break;`
2. BlockCacheManager 无树结构、无 toggleNode
3. 旧版 parsedTree toggle 被新版 shadow

搜索跳转不居中：目标 block 未加载导致骨架屏占位 → 视觉位移。

### 架构诊断结论

当前是**方案 A（扁平化索引）**，三套索引并存（MemoryIndex行号、BlockCacheManager节点ID、parsedTree树形），互不桥接。

### 步骤1：清理残留代码（-196 行）

| 删除内容 | 原用途 | 死因 |
|:---|:---|:---|
| `parsedTree` Worker 变量 | 树形数据结构 | 从未被赋值 |
| `getDisplayValue(node)` | 被 flattenTree 调用 | flattenTree 已死 |

| `collapseAll()` (旧) | 折叠全部（树模式） | parsedTree 永远 null |

### 步骤2：BlockCacheManager 虚拟展开能力

**新增数据结构：**
```
collapsedNodes: Set<number>         — 折叠状态
parentMap: Map<childId, parentId>  — 子→父映射
childrenMap: Map<nodeId, [childId]>— 父→子节点列表
_visibleCache: [...flatNodes]      — 可见节点数组
_visibleIdSet: Set<number>         — 可见节点 O(1) 查找
```

**新增方法：**
| 方法 | 用途 | 复杂度 |
|:---|:---|:---|
| `toggleNode(nodeId)` | 切换折叠，局部重建 _visibleCache | O(子树大小) |
| `_expandInVisibleCache` | 展开：直接子节点插入可见列表 | O(子节点数) |
| `_collapseInVisibleCache` | 折叠：移除所有后代 | O(可见后代数) |
| `_isAncestor` | parentMap 向上追溯 | O(深度) |
| `getTotalVisibleNodes()` | 可见节点总数 | O(1) |
| `getVisibleNodeIndex(nodeId)` | nodeId 在 _visibleCache 中的下标 | O(n) |
| `getVisibleAncestor(nodeId)` | 最近被折叠祖先 | O(深度) |
| `visibleToFlatRange(start, end)` | 可见索引→flatId 范围 | O(1) |
| `prepareJumpToVisible(flatId)` | 自动展开祖先 + 返回可见下标 | O(深度+子树) |

**关键设计**：DFS 顺序保证后代连续 → `_collapseInVisibleCache` 只需一次 `splice(idx+1, count)`。`parentMap/childrenMap` 在 `_buildFlatNodesCache` 的 DFS 中同步构建（零额外遍历）。

### 步骤3：UI 事件流连接

| 变更 | 旧 | 新 |
|:---|:---|:---|

`handleSearchJump(flatId)`：搜索 flatId → prepareJumpToVisible → jumpToLine=visibleIndex → 居中跳转。

## Part 3: 折叠稳定性修复（4 轮迭代）

### 3.1 折叠后回到首行（P0）

**根因**：[JsonTreeView.vue:L154](file:///f:/pwork/project/json-fmt/json-fmt/json-fmt/src/components/JsonTreeView.vue#L154) `watch(totalNodes)` 在 totalNodes 任何变化时无条件 `scrollTop=0`。

**修复**：新增守卫 `prevTotalNodes.value <= 0`，只在文件加载时重置。

### 3.2 折叠后上方大片空白

**根因**：`scrollTop` 像素值不变但 `_visibleCache` 变化，`startIndex` 在新空间指向完全不同节点 → 渲染空段。

**修复**：新增 `flatVisibleStart/End` ref + `syncFlatVisibleRange()` + `jsonTreeRef.setScrollTopToLine()`。

### 3.3 锚点选择纠错

**根因**：锚点用了 `currentGlobalLine`（视口第一行），不是被点击的父节点。

**修复**：锚点改为被点击 `nodeId` 自身。移除 `currentGlobalLine` 逻辑。

### 3.4 移除多余的 scrollTop 调整

**根因**：`_collapseInVisibleCache` 只 `splice(idx+1, count)`，父节点索引不变。`setScrollTopToLine` 反而把节点推到视口顶部。

**修复**：删除 `setScrollTopToLine` 调用。

### 3.5 坐标空间统一（最终修复）

**根因**：`visibleItems = visibleNodes.slice(startIndex - firstId, ...)` 中 `startIndex` 是 visibleIdx 空间，`firstId` 是 flatId 空间——折叠后等价关系破裂。

**修复**：`visibleItems` 改用 flat ID 范围 + 扫描 `visibleNodes` 按 `id` 匹配：

```
新公式: visibleItems = { n in visibleNodes | flatVisibleStart-50 <= n.id <= flatVisibleEnd+50 }
```

**新增链路**：
- `handleNeedNodes` → `visibleToFlatRange` → `flatVisibleStart/End`
- `watch(totalNodes)` → `syncFlatVisibleRange()`（折叠/展开后重映射）
- `:flat-visible-start/end` props 传递到 JsonTreeView

## 已验证方向（防止下次重复验证）

1. **ResizeObserver 在虚拟滚动下有害**——已删除
2. **布尔标志有单次消费缺陷**——改用时间戳窗口
3. **lastRenderedLine 只抑制渲染**——状态机永远执行
4. **displayScrollLine 用 DOM scrollTop**——禁用浮点链
5. **BlockCacheManager onBlocksChanged 需要 needsUpdate 守卫**
6. **Vue scoped CSS 不支持 v-html**——需全局 style
7. **watch(jumpToLine) 值不变不触发**——需 jumpTrigger
8. **overflow-anchor 在虚拟滚动中产生反馈环路**——需 `overflow-anchor:none`
9. **CSS contain:layout style 隔离回流**
10. **折叠后 offsetY 与 startIndex 必须同源**（均用 localScrollTop）
11. **折叠后 scrollTop 不需要主动调整**——父节点在 _visibleCache 中索引不变
12. **visibleItems 不能用 firstId 偏移**——flatId vs visibleIdx 坐标空间不一致

## 关键教训

| 教训 | 说明 |
|:---|:---|
| **坐标空间统一** | 系统存在 flatId 和 visibleIdx 两套空间，折叠前等价，折叠后裂开。visibleItems 公式混用导致空白。最终通过 `flatVisibleStart/End` 统一到 flatId 空间。 |
| **锚点不要干预位置** | `_collapseInVisibleCache` 不改变父节点索引 → 像素位置不变 → 无需调整 scrollTop。干预反而出错。 |
| **层层叠加是坏味道** | 3.1-3.4 每轮都在追加新的 guard/callback/scrollTop调整，3.5 回归本质——坐标空间应始终保持一致。 |
| **先诊断再修改** | 每轮都通过 console.log 或代码走读确认根因后再改，避免无效尝试。 |
